企画・設計段階の振り返り
マーケットリサーチ、ペルソナ設計、情報設計について

【うまくいったこと】
・3C/STPで「浅草×和モダン体験」と「迷わない予約導線」を差別化軸として言語化できた。

・情報設計でトップから各ページへの導線と、予約相談・問い合わせへのCTAを一貫して配置できた。

企画・設計段階では、マーケットリサーチで競合の強みと不足（情報分散・予約導線の弱さ）を整理し、浅草らしい和モダン体験×迷わない予約という差別化軸を定義できた。さらにペルソナの行動から料金・所要時間・体験の流れを最優先に配置し、日英ページでも同じ導線を維持できた。一方で世界観の訴求と即決情報のバランス、ページ階層や名称の切り分けに迷いが出たため、早期に仮説サイトマップを作り検証したい。次回は用語と情報優先度を初期レビューで固定する。

フロントエンド実装の振り返り
HTML/CSS/JavaScript の実装について

【うまくいったこと】
・CSS変数（色/余白/影/角丸）と共通コンポーネントを作り、全ページで統一感を出せた。
・レスポンシブ（container、ブレークポイント、可変幅）を前提に組み、崩れにくいレイアウトにできた。
・JavaScriptはハンバーガーメニューの`aria-expanded`連動、スクロール時ヘッダー状態、アコーディオン開閉、アンカーのスムーススクロールなど、体験向上に直結する箇所へ絞って実装できた。
・問い合わせフォームのバリデーション（必須/メール/選択）とエラー表示、先頭エラーへのフォーカス移動まで一通り作れた。

【苦労したこと】
・`style.css`が肥大化し、ページ別調整が増えるほど見通しが落ちた。
・コンポーネント名や粒度の揺れ（似た見た目の微差）を後から揃えるのに時間がかかった。
・フォーム周りは言語切替（JP/EN）や無効化条件など分岐が多く、テスト観点の洗い出しに手間取った。

HTMLはセクション構造とdata属性を揃え、CSSは変数＋共通コンポーネント化で全ページのトーンを統一できた。JSはヘッダー、メニュー、アコーディオン、フォーム検証を最小依存で実装し、エラー表示とフォーカス移動まで担保できた。一方でCSS肥大化と命名の揺れ、JSの責務集中が課題。次回は設計ルールを固定し、CSS分割とユーティリティ整理、JSのモジュール化で保守性を上げたい。

バックエンド実装の振り返り

【うまくいったこと】

・Flywayでテーブル作成/変更をマイグレーションとして残し、H2（ファイルDB）でローカル実行しやすくできた。

【苦労したこと】
・JPAとMyBatisを併用しているため、永続化の流れや責務の切り分けを整理するのに時間がかかった。
・問い合わせフォームは日本語/英語、問い合わせ種別による必須項目の分岐が多く、バリデーション設計が複雑になった。
・論理削除（deletedAt）対応により、取得条件や表示順、管理画面の導線を揃える調整が必要だった。

Spring Securityで`/admin/**`を保護し、問い合わせはJPAで保存〜一覧/詳細/編集/論理削除/復元まで実装できた。日付の正規化や選択肢コード検証も入れ、入力の揺れに対応できた。FlywayでDB変更履歴を追える点や、管理者ユーザーのシードも運用面で助かった。一方、JPA/MyBatis併用とJP/EN分岐で設計が複雑化したため、次回は永続化手段の統一とバリデーション/メッセージの共通化で保守性を上げたい。

2 技術記事

技術記事タイトル

Spring Securityで管理画面ログイン（/admin/**）を実装する（フォーム認証＋ロール制御）

記事本文
上記の構成に沿って記事本文を記載。コードブロックを含めてください。

1. はじめに（何をしたかったか）
ポートフォリオサイトに問い合わせ一覧やユーザー管理などの管理画面を作ると、URL直打ちで誰でも見えてしまうのは危険です。そこで「公開ページは誰でも閲覧OK」「管理画面（/admin/**）だけログイン必須」にし、さらに管理者だけが操作できるようにしました。
※用語：認証＝ログインして本人確認すること／認可＝その人に操作を許可すること。

2. 課題（何が難しかったか）
Spring Securityは安全側がデフォルトなので、設定が曖昧だと意図しないURLまでブロックされがちです。また、AIにプロンプトを投げても違う意図の対応をされることがあり、実装方針がぶれて手戻りが出ました。

3. 調査（どのように調べたか）
まず「公開するURL」と「守るURL」を紙に書いてから、最小構成で動く状態を作りました。確認観点は「未ログインで/admin/**に入るとログインへ誘導されるか」「ログイン失敗が画面で分かるか」「ログアウトでセッションが切れるか」です。プロンプトは、指示を細かく出し、禁止事項（推測で書かない・対象URL/ファイルを限定する等）を入れて精度を上げました。

4. 実装（コード付きで解説）

4-1. /admin/** をログイン必須にする（認証）
下の設定で、公開ページ（/ や /css/** など）は許可し、/admin/** はログイン必須にします。
※セッション＝ログイン状態をサーバー側で覚える仕組み。

// SecurityConfig.java（抜粋）: URLごとの許可/禁止とフォームログイン設定
http.authorizeHttpRequests(auth -> auth
    // 公開：サイト本文や静的ファイル
    .requestMatchers(
        "/", "/index.html",
        "/about.html", "/access.html", "/skills.html", "/works.html", "/contact.html",
        "/en/**", "/css/**", "/js/**", "/images/**"
    ).permitAll()
    // 公開：ログイン画面は誰でも表示できる必要がある
    .requestMatchers("/admin/login").permitAll()
    // 保護：管理画面配下はログイン必須
    .requestMatchers("/admin/**").authenticated()
    .anyRequest().permitAll()
)
.formLogin(form -> form
    // ログイン画面（GET）とログイン処理（POST）のURLを揃える
    .loginPage("/admin/login")
    .loginProcessingUrl("/admin/login")
    // フォームのname属性に合わせる
    .usernameParameter("username")
    .passwordParameter("password")
    // 失敗時に ?error を付けて戻す（画面で表示しやすい）
    .failureUrl("/admin/login?error")
)
.logout(logout -> logout
    // ログアウトでセッションを破棄してログイン画面へ
    .logoutUrl("/admin/logout")
    .logoutSuccessUrl("/admin/login?logout")
    .invalidateHttpSession(true)
    .deleteCookies("JSESSIONID")
);

4-2. ログイン画面で失敗/ログアウトを見せる
ログイン失敗（?error）やログアウト後（?logout）を画面に渡し、「何が起きたか分からない」を減らします。

// AdminAuthController.java（抜粋）: /admin/login を表示する
@GetMapping("/admin/login")
public String login(
    @RequestParam(value = "error", required = false) String error,   // // ?error が付くと失敗
    @RequestParam(value = "logout", required = false) String logout, // // ?logout が付くとログアウト後
    @ModelAttribute("loginForm") LoginForm loginForm,
    Model model
) {
  model.addAttribute("loginError", error != null); // // 画面でエラーメッセージを出すフラグ
  model.addAttribute("loggedOut", logout != null); // // ログアウト完了の表示用
  return "admin/login";
}

4-3. 管理者だけ操作できるようにする（認可）
URLで/admin/**を守るだけだと「ログインできる人全員」が操作できる可能性があります。そこでControllerにロール（権限）を要求します。
※ロール＝ユーザーの役割（例：ADMIN/USER）で、できる操作を分ける考え方。

// AdminUserController.java（抜粋）: 管理者だけ許可
@RequestMapping("/admin/users")
@PreAuthorize("hasRole('ADMIN')") // // ROLE_ADMIN を持つユーザーのみアクセス可
public class AdminUserController {
  // ...
}

4-4. DBのユーザーをSpring Securityへ渡す
Spring Securityは、ログイン時に「ユーザー名からユーザー情報を取得する処理」が必要です。UserDetailsServiceでDB検索し、ROLE_プレフィックス付きの権限を渡します。

// AdminUserDetailsService.java（抜粋）: DBユーザーを認証に利用する
public UserDetails loadUserByUsername(String username) {
  UserAccount user = userMapper.selectByUsername(username); // // DBから検索
  if (user == null) {
    throw new UsernameNotFoundException(username); // // 見つからなければログイン失敗
  }
  return new User(
      user.getUsername(),
      user.getPasswordHash(), // // パスワードは平文ではなくハッシュ（暗号化のようなもの）で保持
      user.isEnabled(),
      true, true, true,
      AuthorityUtils.createAuthorityList("ROLE_" + user.getRole()) // // 例: ROLE_ADMIN
  );
}

5. 結果（どう解決したか）
未ログインで/admin/**にアクセスするとログイン画面に誘導され、ログイン成功後は管理画面を利用できます。さらに@PreAuthorizeで管理者専用機能を縛り、ログアウト時はセッション破棄まで行えるようになりました。プロンプト面では、指示を細かくし禁止事項を入れることで、意図ズレによる手戻りを減らせました。

6. 学び（何を得たか）
Spring Securityは「どこを公開し、どこを守るか」を先に設計してから実装すると理解が進みます。URL制御（/admin/**）と機能制御（@PreAuthorize）を分けると、将来の追加機能でも漏れが起きにくいです。プロンプトは“曖昧さがそのまま出力のブレになる”ので、目的・対象・制約・禁止事項をセットで書くのが効果的でした。

【短期的に学ぶべきこと（3ヶ月以内）】
Spring Boot/Javaの設計（Controller/Service/Repositoryの責務分離、例外設計、@Valid＋独自検証の共通化、エラーメッセージ整理）を固める。Spring Securityは/admin配下の認証・認可（URL/メソッド）、セッション、CSRF、ログアウト挙動を整理。DBはFlyway運用前提で正規化と変更手順を復習。CSS/JSは分割・命名統一で保守性を上げ、Gitは1コミット1目的を徹底。

【中期的に学ぶべきこと（半年〜1年）】
JUnit＋Springテスト（Web/Service/Security）で品質を強化し、変更に強い構成へ。DBはトランザクション/インデックスまで広げ、移行・正規化の判断力をつける。UI/UXとFigmaは状態設計（成功/失敗/空/権限なし）と部品表で再現性を高める。プロンプトはテンプレ化して指示品質を安定化。

【推奨する学習リソース】
Spring公式（Boot/Security/JPA）、Flyway公式、SQL入門書、JUnit入門書。作成済みの/admin認証・問い合わせ管理を小さく改善しながら学ぶ。
